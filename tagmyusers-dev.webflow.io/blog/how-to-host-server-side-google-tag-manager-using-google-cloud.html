How to set up server-side Google Tag Manager using Google Cloud

Setting up a Google Tag Manager (GTM) Server-Side container is an essential step for marketers looking to enhance data security, improve site performance, and gain greater control over their tracking setup. 

This guide is designed to walk you through the entire process, from configuring your Google Cloud Platform (GCP) project to deploying and verifying your GTM Server-Side container. 

Whether you're a beginner with a basic understanding of client-side GTM tracking or looking to advance your tagging strategy, these step-by-step instructions will help ensure a smooth transition to a more secure and efficient server-side tagging environment.

Why is server-side tagging so important? 

With Google launching consent mode v2 (Google’s latest consent mode policy) and other stricter regulations, seasoned marketers are already shifting towards the server-side tagging approach.

The traditional tagging method is just not enough, especially for larger organizations, to get accurate data in their analytics platforms.

But before I get into the core benefits of server-side tagging, let me give you a recap of how the traditional approach works and how server-side tagging differs from it.

The traditional approach of using GTM

 The traditional approach goes like this:

  1. You deploy the web container’s javascript snippet into the source code of your website. 
  2. You set up tags provided by different analytics tools like GA4, Facebook Pixel, etc for tracking a specific metric
  3. A user visits your website. The browser loads and fires different snippets/scripts/tags based on user behavior (e.g. scroll depth, form submission, etc.)
  4. The script then sends this information to related advertising/analytics tools like GA4 or Google Ads where you can perform your analysis

All of this happens on the client side or in simple words, within the user’s browser. This approach works until it faces an adblock extension or cookie consent issue. More on this later in the guide.

The server-side approach

Enter server-side tagging. Here is how it works:

  1. You set up a server-side container in Google Tag Manager (Often using Google Cloud). This container will handle all your tracking and analytics needs
  2. When a user interacts with your website, the data like page view, scroll depth, etc., is first captured by GTM’s client-side container in the browser. This information then gets passed on to your server-side container.
  3. The GTM server then processes the data, enhances it if needed, and even adds more details that are not available on the client side.
  4. After processing, the server container sends the cleaned and structured data to connected analytics/advertising platforms like GA4, Google Ads, etc.

This process is basically a mix of client-side and server-side systems that work hand in hand to produce the best output.

You can go even deeper by skipping the client-side container altogether and sending data from one server to the GTM server-side container. But that is a discussion for another day as it requires complex coding and APIs.

What are the benefits of server-side tracking?

Improve your site speed

When a site loads, it has to process a lot of codes and download a lot of resources including tracking snippets you add for analytics purposes.

The more snippets, the worse the site speed - something actually proven by avid marketers and endorsed by Google itself.

Solution? Server-side tagging. You are effectively replacing all those Javascript snippets with one single piece of code that will handle all your analytics needs.

So if you run a big business and have a lot of metrics to track, you will be better off with implementing a server-side GTM container.

Reduce the impact of ad blockers

Ad blockers are the number one enemy of tracking systems. They essentially block most of the requests directed towards tracking platforms resulting in you missing out on absolutely important user behavior data needed to make accurate decisions.

And when you get to know about a study that says almost 40% of the world uses ad blockers, it makes you worried. Server-side tracking ensures you bypass those ad blockers and receive proper data in your analytics platforms.

Enhances privacy and data control

Chrome as well as other popular search engines are phasing out third-party cookies. This means that client-side tracking will slowly lose its effectiveness as it is highly dependent upon third-party cookies.

Server-side tracking, on the other hand, addresses this issue by using privacy-oriented first-party cookies. Meaning, it sends data directly back to the website’s server instead of multiple third-party vendors.

Comply with data protection regulations

You may not know that but laws around data protection (GDPR, CCPA, etc) are quite strict and can get you in hot waters if you do not follow them religiously. 

It is imperative for a business to make sure they do not collect and send a user’s (personally identifiable information) to third-party vendors as it can get them in serious legal trouble. And if you are responsible for their marketing efforts, this responsibility falls 100% on you.

Does client-side tracking give you this control? I’m afraid not. Server-side tracking is the only way to get 100% control over your data and filter out any and all sensitive information.

Create your GTM server-side container

First things first, configuring GTM server-side tracking can be a little tricky. It takes a good understanding of how tracking works on your website.

But I’m going to make it as simple as possible. You can consider this guide as an SOP designed specifically for a marketer implementing the GTM server-side container for the first time.

With that being said, let’s get into the nitty-gritty of this process.

Go to “tagmanager.google.com,” log in, and create a new account.

Add “Account Name” and “Country” information. You can use your company’s name here. As this is a tutorial, I’m going to name my account “TutorialGTM”.

In “Container Setup,” add your website domain and select “Server”.

Hit “Create” and agree to Google Tag Manager’s terms and conditions.

Setting up your tagging server

Once inside your container, the tag manager will ask you to set up your tagging server.

You can either go the manual route (which you should if you already have a server setup) or you can go the automatic route through Google Cloud Platform.

For this guide, we’ll go the automatic route.

Next, it’ll ask you to create a billing account. Go ahead and click on “Create a billing account”

Then, it will redirect you to Google Cloud Platform so that you can set up your payment profile.

Follow the platform’s instructions and come back when your billing account is ready.

After a few seconds of loading time (sometimes it can take a few minutes), your tagging server will be created automatically and you’ll be able to see the information about the created server (container configuration, project ID, default URL, etc.)

Configuring tags, triggers, variables and clients 

Just like the traditional approach, you’ll have to configure tags, triggers, and variables to set up conversion tracking. The only new thing is setting up “Clients.” 

A “Client” basically receives the data from the GTM server-side container and makes it available to tags and triggers in the client-side container for further use.

You get a GA4 client by default after setting up your tagging server that you can use for GA4-related conversion tracking. 

You are free to create more clients according to your needs and the tracking scripts that you use. If you are not an expert yourself, it’d be better to find one and delegate this task.

As you are just starting out, it’d be better to just stick to the default GA4 client and learn how to use it.

So, when GA4 sends data to your GTM server, this client will process it before sending it to tags and triggers.

Now that the client is configured, it’s time to set up a GA4 server tag. Its purpose is to relay the received information to GA4 servers/endpoints.

First, go to “Triggers” and click on “New” We need to create a custom trigger first which is configured to trigger whenever the “Client Name” contains “GA4.”

Next, under “Trigger Configuration,” select “Custom”

Click on “Some Events” and set the condition to be: Client Name > Contains > GA4.

Rename the trigger to something like “Custom Trigger: GA4” and hit “Save”.

Next, go to “Tags” and click “New”.

Under “Tag Configuration”, select “Google Analytics: GA4”

Under “Triggering,” choose “Custom Trigger: GA4”

Rename the tag to something like “GA4 Server Tag” and hit “Save”.

With this done, it is time to configure your client-side “Google Tag” so that it connects to your server-side “GA4” tag.

Sending GA4 data from client-side to server-side

There are several ways to do this. A developer can place a gtag.js snippet code into your site’s source code. Or you can have a custom code or library to handle it for you. 

But we are going to take the simplest route as I promised to keep this guide simple. We are going to use your regular GTM container (I suppose you already have one if you are looking into server-side.) 

Go to “Tags” under your regular GTM container and open the tag you set up to configure Google Tag with your website. We are going to edit it such that it starts relaying the info to your GA4 server tag instead of directly to the Google Analytics platform.

Under “Tag Configuration”, go to “Configuration Settings” and edit the fields as shown below:

In the “Value” field, paste your “Default URL” that you got just after installing the tagging server.

Once everything is set up, rename the tag to something like “G-Tag Server” and hit “Save”.

Next time you set up any GA4 event tag, make sure you add “server_container_url” under the “Event Parameters” section. 

With this step, you have successfully set up server-side tracking through GTM. To make sure it all works, let’s put it to test using the “Preview” mode.

How to test your server-side GTM container

First, go back to your server-side GTM container and hit “Preview”.

This will trigger preview mode, an interface that helps you debug, diagnose and validate your Google Tag Manager setup.

You should run a brief audit using preview mode whenever you make changes to your GTM configuration.

Once the preview mode opens, keep it there and move back to your regular GTM container.

Now, open preview mode in your regular GTM container and see if the Google Tag fires (it should if you set it to fire on all pages.)

If it did, you should be able to see the “GA4 Client” fired on the server-side preview mode.

This verifies that you've followed all of my instructions and done a great job - Congratulations!

Frequently asked questions

How expensive is Google Cloud? 

Google Cloud Platform’s pricing strategy is based on the “pay-as-you-go” pricing model. You only pay for the services you use. 

In this specific scenario, a single instance can cost you nearly $100/month given that the traffic to your website isn’t very high. If the traffic is high and Google has to deploy more instances, your monthly price can easily go up to $180 - $200 per month.

There is no way to calculate the exact cost until you start using your tagging server setup.

Are there any alternatives for server-side GTM hosting? 

If the Google Cloud Platform feels too advanced for you, you can always consider other custom hosting solutions like TagMyUsers. We offer managed server-side container hosting for GTM and Facebook CAPI. You’ll get a production-ready server-side container in just one click.

How can I make sure my server-side container is running? 

You can check if your server-side container is running using the “Preview” mode. Follow these steps:

  • In your GTM Server-Side container, click on "Preview."
  • This will open a new tab with the GTM debugger interface for your server container.
  • From your client-side GTM container, send a test event to the server-side container. You can do this by triggering any event that is configured to send data to the server-side container.
  • Check the GTM debugger interface to see if the request is received and processed correctly.
