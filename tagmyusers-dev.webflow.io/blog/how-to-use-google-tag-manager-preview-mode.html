How to use Google Tag Manager Preview Mode to debug your tracking

Ever found yourself scratching your head over why your tag isn’t firing? Or maybe you're wondering what kind of data you can send to Google Analytics, or why a trigger didn’t activate? If you've been there, you're not alone.

When using GTM, issues often arise that require deeper investigation. Debugging is frequently overlooked, however it is used to test the user experience and ensure data integrity.

Discover how to make the best use of Google Tag Manager’s Preview Mode (or Preview and Debug mode) with the help of our guide. 

What is Google Tag Manager Preview Mode?

Google Tag Manager Preview mode (P&D) is a tool that lets you access a site where GTM container code is installed and identify which tags fire, what information they pass to analytics/marketing platforms, and so on. It will not negatively impact your actual data. 

In Google Tag Manager's preview mode, you get three browser tabs (2 tabs and an additional window):

The GTM management interface

The GTM preview tool

The tagging assistant

Why should you use preview mode?

GTM’s preview mode helps you:

  • Find out which tags have fired on specific website interactions.
  • List down the tags that did not fire and the reason.
  • See details on all the variables and values available on the page.
  • Observe any alterations of the Data Layer during the interaction

It is advisable to test your changes before publishing to avoid distributions in your analytics setup.

Is it the same as debug mode?

Yes, Preview Mode is referred to by a different name: Debug Mode. Both are synonyms for each other and mean the same thing—it provides a controlled environment in which you can view how your tags are working.

Now, let’s explore how the preview mode operates and how to use it.

How can I enable preview and debug mode?

Step 1: Click the preview button

To allow the Debug mode in the Google Tag Manager, go to the upper panel of the GTM interface. Click the Preview button at the right corner of the Submit button.

Step 2: Launch Tag Assistant

After you click the Preview button, a new browser will open with tagassistant.google.com

Step 3: Enter the URL

After downloading, a popup window will appear on your screen. Enter the address of the page that needs to be tested and debugged. It would be your homepage or specific URL. Enter the URL and click Connect.

Step 4: View the Debug Page

A new tab or browser window displays the URL you entered. If the page is not loading correctly, try re-enabling preview mode and unchecking the corresponding checkbox in the preview popup.

Go back to the tab in which you have open tagassistant.google.com and look for a success message as follows:

Refreshing after container changes

If you made changes to your container and you wish to see those changes on your website, then you need to refresh the preview mode.

To do this, navigate to the GTM interface and click the Preview button again. Follow the steps for reloading the preview mode.

Understanding the GTM preview mode interface

The GTM Preview Mode interface has several key sections that provide detailed insights into your tags and triggers:

Event Timeline

An event timeline shows all the events on your site, for instance, page views, form submissions, and clicks. It assists you in identifying when a specific tag is triggered.

Tags

The tags appear in this section with a specification about which of the tags fired for a given event and which did not and their reasons.

Variables

The preview uses variables to capture dynamic values. This section makes it easier to see the actual values of variables in different events.

DataLayer

This section denotes the state of the data layer at a point in time and assists in debugging issues where data is pushed into the data layer.

Consent

You can see the consent status here if you have used any consent management platforms and how it impacts tag firing.

Errors

If there are any errors made while executing tags, they will appear here. It will be easier for you to spot the mistake.

Header

Here you can observe the connection status of the Preview mode with the newly opened website window. If you’re using Google Analytics version gtag.js or GA4, you will see a dropdown list to toggle between them.

Container Details

This section contains information regarding the version of the container you are previewing, the version number, and the date of the last update.

Debugging events

In the Preview and Debug console, the left side contains all events in the data layer. Each page load must show five events. These events include;

  • Consent Initialization
  • Initialization
  • Container Loaded (earlier called Pageview)
  • DOM ready
  • Window loaded

Additional events may also occur, such as Message, which is okay. However, five of these must be included on every page. If there is no Container Loaded event, look for a broken data layer.

Also, during the debugging process, if you are moving from one page of your website to the other, all the events that are put in preview mode will be grouped according to the page.

The Five Events on Every Page

  1. Consent Initialization (gtm.init_consent):

It is when consent-related settings (such as a cookie consent banner or Google Consent Mode tag) should be loaded.

  1. Initialization (gtm.init):

It is when main tracking codes that load JavaScript libraries such as the Google Tag (Google Config Tag) should be fired.

  1. Container Loaded (gtm.js):

It is the time to fire all pageview-related tags that are not configuration tags. It used to be the first event in GTM, but now it occurs after Consent Initialization and Initialization.

  1. DOM Ready (gtm.dom): 

This event takes place when the HTML of the website is downloaded, the document is built and the users begin to view elements on the site.

  1. Window Loaded (gtm.load):

This event is fired when everything, including JavaScript, has loaded.

Additional events in Google Tag Manager

GTM offers several pre-built triggers: page view (such as Page view, Window Loaded, and DOM ready), Click ( such as link or element click), Form Submissions, Timers, History Changes, JavaScript Errors, and more.

Once triggered, it observes activity on your site. For instance, the Form Submission triggers when forms are filled and submitted. The Link Click trigger looks at the events when visitors click particular links.

These interactions show up on the Event Timeline when they take place. Custom events are those events you, developers, or the third-party solutions integrate into your site.

In the Tags tab, it is clear which tags are fired when an event in the timeline is selected and which are not, so it is essential for debugging and checking the performance of tags

Debugging tags

It displays all the available tags, divided into two groups:

  • The groups that targeted the selected event.
  • The groups that did not target the selected event.

It provides information on why a specific tag triggered or did not trigger. Go to the Event Timeline and select any event. Click on the tag of your choice.  

You will get a breakdown of the tag details,

  • Properties of the Tag
  • Triggers of the Tag
  • Blocking Triggers

When you use multiple and complex triggers, you can also check the status of the individual condition. The green checkmark means the condition has been met and the red cross means that it has not.

Near the top right-hand corner of the preview mode, there are buttons which will toggle between ‘Names’ and ‘Values’ of variables.

  • “Names” demonstrates the names of the variables.
  • “Values” displays the value of the variable.

Using “Values” is more comfortable when it comes to defining which data goes to analytics and marketing tools and whether the trigger values meet the criteria.

Never debug individual tags while the Summary option is chosen above the event timeline. However, in the event stream always choose the event and then click on the Tag.

Debugging variables

Variables include the event where the variable is in, the variable type, the type of data that is returned, and the resolved value. The variables tab contains information on the selected variable including variable type, data type of the returned data, and resolved value.

These variables are universal and can be placed in any tag or trigger of your choice, Mixpanel event tags, Google Analytics event tags, or other variables. To include a variable, surround it with curly brackets {{ }}.

Navigate between GTM events in the timeline to check how a variable’s value may differ based on the context.

Using the data layer

The Data Layer tab is underestimated by beginners and even many intermediate users of GTM. This tab provides the actual message object sent to the data layer for the chosen event and shows the state of the data layer when the message transaction is done.  

You can view all the available data in the data layer and the values after the execution of a specific tag in the Google Tag Manager. The best feature is that any data point that is sent to the data layer can be made into a GTM variable.

The Variables tab under the Debugging pane in Google Tag Manager only lists variables configured in the GTM interface like Page Path and Page URL. You can only use the information from the data layer in Google Analytics if you create the data layer variable through the GTM interface.

For instance, if you would like to track a Google Analytics event when a comment is left and add the full name of the author of the article, then you have to create a data layer variable.

On your GTM container, navigate to Variables and add a new one as desired (dlv is abbreviated as data layer variable).

There should now be a new variable pop-up under the Variables tab each time you refresh the P&D mode and your website. You can then use that {{dlv – Post Author}} variable in the Google Analytics or other tags.

Auditing consent states

This tab indicates the consent status if Google Consent Mode is installed for example; denied or granted.

This can also result to an error if, for instance, the tag will try to read the consent status when the consent mode has not been initiated yet.

Finding errors

This tab provides information on errors if a GTM Tag Template does not fire. If you find any other number excluding 0 at any of these places, click on it to know the cause.

This tab is helpful for problem-solving and can help you locate issues before they become a problem. You should get in the habit of frequently checking it.

How can I refresh the preview mode?

In order to see the modified changes in the container on your web site, it is necessary to refresh the preview mode.

Visit the GTM interface, click on "Preview" again and follow the previous steps to start a new preview session.

How can I share my debug session?

If you need to share your debug session with others, click the "Share Preview" button in GTM. It creates a URL that others can use to view the debug information; it facilitates collaboration on the same information.

Enabling and Sharing Preview Mode

  • First, turn the preview mode on yourself.
  • Navigate to the top right corner of your GTM interface and click the three dots accompanied by the Share option.

Entering your website URL and container celection

  • Provide a link to the website where you want to enable the preview mode.
  • Select the GTM container from the drop-down list (if there are many containers).

‍Managing your shared previews

To manage shared previews:

  • Return to tagassistant.google.com
  • Identify all the available and enabled preview modes.
  • Click on the three dots next to the preview you want to share. Then click Share.

Accessing previously enabled previews

If you have enabled the preview mode earlier and want to do it again, visit tagassistant.google.com. Follow the procedure above to share it over again.

Exiting the preview mode

To exit the preview mode, click on Finish in the preview mode badge located at the lower right-hand corner of the web page.

Can you debug multiple domains with the same container?

When relying on several domains that are associated with a single Google Tag Manager container, start the preview mode on both domains.

  • Open tagassistant.google.com in the Google Tag Manager Interface. 
  • Press the Preview icon and fill in the domain of the second site, following the steps to enable the preview mode.
  • Next, on tagassistant.google.com, at the top left, click on the X icon, then from the dropdown choose Keep the domain XXXX enabled for debugging, then click on Close Debugger. It will keep the second website in preview mode.
  • Then click on either the Preview button in the Google Tag Manager interface or the Add domain button in tagassistant.google.com and attach the domain of the first website. Complete all actions before Website 1 is displayed in the new window or tab.

This process sets the first-party cookie _TAG_ASSISTANT=X on both domains. It enables you to browse through both domains in the preview mode with all events or data of both websites.

Improvements with the Tag Assistant Chrome extension

To get more from Tag Assistant, especially with the Google Chrome browser, you can use the Tag Assistant Companion extension. This extension is useful for enabling the Preview mode as it;

  • Open your website in a new tab instead of a new window. 
  • Allows debugging multiple tabs simultaneously. You can debug on two tabs at once.
  • Adds shapes and emojis. These visual aids assist you in easily pointing out the window that is being debugged at any particular time.

Excluding tag manager traffic from your analytics

If you observe that there is traffic coming from tagassistant.google.com or you are seeing URLs with gtm_debug=x in your GA reports, you should remove this traffic. To do this;

  • Navigate to Admin> Property> Data Streams> Choose Web> Tag Settings> Show All> List unwanted referrals.
  • Add tagassistant.google.com.

Practical Tips for Using GTM Preview Mode

Understand the event order in the debug console

When you’re in the GTM preview mode, it’s crucial to understand the event order displayed in the Event Timeline.

If you notice the words “Message” before “Page view,” this indicates that some data was pushed to the Data Layer without the “event” key. The next step would be to use the GTM debug mode to investigate further.

Use clear naming conventions

To make debugging easier, always use clear and consistent naming conventions for your tags, variables, and triggers.

This practice not only helps you but also anyone else working on the setup to quickly understand what each element does.

Document your changes

Keep a detailed record of any changes you make to your GTM setup. This documentation will be invaluable for tracking problems and troubleshooting issues that arise.

Regular updates to this record ensure you have a reliable history of modifications.

Get in the habit of testing regularly

Regular testing is key to maintaining a robust GTM setup. Make a habit of using the preview mode often, especially after making significant changes to your tags.

This proactive approach helps catch and fix issues before they impact your live environment.
